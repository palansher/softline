# 📋 Памятка: Анатомия нового проекта Django

## Моя структура

```text
.
└── lesson33
    ├── db.sqlite3
    ├── lesson33
    │   ├── asgi.py
    │   ├── __init__.py
    │   ├── __pycache__
    │   ├── settings.py
    │   ├── urls.py
    │   └── wsgi.py
    ├── manage.py
    └── my_app
        ├── admin.py
        ├── apps.py
        ├── __init__.py
        ├── migrations
        ├── models.py
        ├── __pycache__
        ├── templates
        ├── tests.py
        └── views.py

```

В**нешняя папка lesson33** (Корневой каталог проекта): Это просто папка-контейнер для твоего проекта. Она может называться как угодно — её имя вообще не влияет на код. Внутри неё лежит файл manage.py.

**Внутренняя папка lesson33** (Главный модуль/конфигурация): А вот это уже настоящий Python-пакет. Внутри него лежат настройки всего проекта (settings.py), маршруты (urls.py) и точка входа для сервера (wsgi.py/asgi.py).

>Совет: Чтобы не путаться, внешнюю папку можно смело переименовать вручную, например, в my_project_folder. Внутреннюю папку трогать нельзя, так как её имя прописано в настройках.

## 🏢 1. Корневой уровень (Уровень управления)

* **`manage.py`**
* **Зачем нужен:** Твой главный пульт управления проектом через командную строку.
* **Что делает:** Ты будешь использовать его постоянно для запуска сервера разработки (`runserver`), создания таблиц в базе данных (`migrate`), создания новых приложений (`startapp`) и создания суперпользователя (`createsuperuser`). Внутри него написан код, который просто связывает твои команды с библиотекой Django. *Код в этом файле менять не нужно.*

## 🧠 2. Папка конфигурации проекта (`lesson33/lesson33/`)

Это «мозг» твоего сайта. Здесь настраиваются глобальные правила игры.

* **`__init__.py`**
* **Зачем нужен:** Это пустой файл, который говорит Python: *«Эй, эта папка — не просто набор файлов, это Python-пакет (модуль), из него можно импортировать код»*. Благодаря ему ты можешь писать, например, `from lesson33 import settings`.

* **`settings.py`**
* **Зачем нужен:** Главный конфигурационный файл всего сайта.
* **Что внутри:** Здесь подключается база данных (по умолчанию SQLite), регистрируются новые приложения в списке `INSTALLED_APPS`, настраивается часовой пояс, язык сайта, пути к файлам стилей (CSS) и картинокам, а также хранятся секретные ключи безопасности (`SECRET_KEY`).

* **`urls.py`**
* **Зачем нужен:** Маршрутизатор или «карта» твоего сайта.
* **Что делает:** Когда пользователь заходит на `[mysite.com/contacts/](https://mysite.com/contacts/)`, Django смотрит именно в этот файл. Он ищет адрес `/contacts/` в списке и решает, какую именно логику (функцию из приложения) запустить, чтобы показать пользователю нужную страницу.

* **`wsgi.py`**
* **Зачем нужен:** Точка входа для традиционных веб-серверов при запуске живого сайта в интернете.

`wsgi.py` — это, по сути, «дверь» для веб-сервера в твое Django-приложение.

Когда пользователь в браузере вводит адрес твоего сайта, этот запрос сначала попадает на полноценный веб-сервер (например, Nginx или Apache), который установлен на сервере (хостинге). Но Nginx сам по себе не умеет читать код на Python. Ему нужен посредник, который переведет запрос с языка веб-сервера на язык Python.

Этот стандарт-посредник и называется **WSGI** (Web Server Gateway Interface). А файл `wsgi.py` — это стандартный шлюз, который берет входящий запрос от веб-сервера и передает его внутрь Django, а потом возвращает ответ обратно.

*Примечание:* Когда ты запускаешь `python manage.py runserver`, Django включает свой простенький, встроенный сервер для разработки. Ему `wsgi.py` не нужен. Но как только ты решишь выложить сайт в интернет («в продакшн»), без `wsgi.py` ничего работать не будет.

* **`asgi.py`**
* **Зачем нужен:** Более современный брат-близнец файла `wsgi.py`.
* **Что делает:** Используется для **асинхронных** веб-серверов. Если на твоем сайте планируются чаты в реальном времени, уведомления без перезагрузки страницы или работа с технологией WebSockets, веб-сервер будет общаться с Django через этот файл.

## ⚙️ 3. Папка приложения (`my_app/`)

Это изолированный блок, где пишется конкретный функционал. Таких папок в проекте может быть много.

* **`migrations/` (каталог)**
* **Зачем нужен:** Папка, где Django хранит историю изменений твоей базы данных. Когда ты создаешь или меняешь структуру таблиц в `models.py`, Django генерирует в этой папке файлы-инструкции, чтобы база данных знала, как ей обновиться.

* **`admin.py`**
* **Зачем нужен:** Настройка встроенной панели администратора Django. Здесь ты указываешь, какие таблицы и данные ты хочешь видеть и редактировать через удобную админку (`[mysite.com/admin/](https://mysite.com/admin/)`).

* **`apps.py`**
* **Зачем нужен:** Файл с системными настройками самого приложения (например, здесь задается его техническое имя). Обычно его редактируют редко.

* **`models.py`**
* **Зачем нужен:** Архитектура твоей базы данных.
* **Что делает:** Вместо написания сложных SQL-запросов, ты описываешь таблицы с помощью обычных классов Python. Django сам превратит этот код в таблицы в базе данных.

* **`tests.py`**
* **Зачем нужен:** Файл для написания автоматических тестов, которые проверяют, что твой код работает правильно и не ломается при изменениях.

* **`views.py`**
* **Зачем нужен:** Твоя главная рабочая зона — логика (контроллеры).
* **Что делает:** Здесь ты пишешь функции или классы, которые обрабатывают запросы пользователя. Например: пользователь запросил страницу товара -> функция в `views.py` берет товар из базы данных, берет нужный HTML-шаблон, соединяет их и отдает пользователю.
