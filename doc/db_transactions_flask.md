# Взаимодействие Flask и PostgreSQL (psycopg2)

**Скрытые транзакции, риски блокировок и методы защиты**

При проектировании веб-приложений на Flask с использованием низкоуровневого драйвера `psycopg2` разработчики часто сталкиваются с зависанием базы данных, падением скриптов миграций (DDL) по таймауту и утечками памяти в PostgreSQL.

Причина этого — фундаментальный конфликт между стандартом Python DB-API 2.0 и многопоточной архитектурой веб-серверов. В данной статье подробно разобраны механизмы неявных транзакций, риски их использования в вебе и три архитектурных подхода к решению этой проблемы.

---

## 1. Корень проблемы: Архитектура неявных транзакций в `psycopg2`

Согласно спецификации PEP 249 (Python DB-API 2.0), которой строго следует драйвер `psycopg2`, управление транзакциями по умолчанию является **неявным (Implicit Transactions)**.

### Как это работает под капотом

#### 1

Как только Flask-приложение выполняет **любой** первый SQL-запрос (даже самый безобидный `SELECT * FROM catalog` для чтения витрины), `psycopg2` автоматически отправляет в PostgreSQL скрытую команду **`BEGIN`**.

#### 2

Открывается транзакция. С этого момента база данных ожидает от Python-кода явного завершения этой транзакции командами:

* `connection.commit()` — зафиксировать изменения.
* `connection.rollback()` — откатить изменения.
  
#### 3

Сам по себе `psycopg2` **никогда автоматически не закрывает транзакцию**.

### Конфликт с веб-средой

В обычном скрипте автоматизации процесс завершается за секунду, и транзакция закрывается вместе с закрытием скрипта. Но Flask — это постоянно запущенный веб-сервер. Когда роут отдал HTML-страницу пользователю, сетевое соединение с базой данных (Connection) возвращается в пул или остается висеть в памяти. Но транзакция на нем **остается открытой**!

В мониторинге PostgreSQL (`pg_stat_activity`) такое соединение получает критический статус **`idle in transaction`** (простаивает в транзакции).

---

## 2. Главные риски зависших транзакций (`idle in transaction`)

Длительное нахождение соединения в состоянии `idle in transaction` влечет за собой три серьезные проблемы:

### А. Блокировки таблиц и DDL-скриптов (Locks)

Любая открытая транзакция накладывает на задействованные таблицы блокировку (как минимум, `RowShareLock` при `SELECT`). Если в этот момент разработчик попытается запустить SQL-скрипт обновления структуры (например, `DROP TABLE`, `ALTER TABLE` или `DROP VIEW`), СУБД заблокирует этот скрипт. Он будет бесконечно висеть в ожидании освобождения таблицы, пока клиент не отвалится по таймауту с ошибкой в логах Postgres:
`LOG: unexpected EOF on client connection with an open transaction`.

### Б. Деградация производительности и раздувание БД (Bloat)

PostgreSQL использует механизм MVCC (многоверсионность строк). Когда строка обновляется или удаляется, старая версия физически остается на диске. Специальный фоновый процесс `VACUUM` очищает этот «мусор». Однако `VACUUM` не имеет права удалять старые версии данных, если они были созданы *до* начала самой старой из всё ещё открытых транзакций в системе. Одна забытая транзакция от `SELECT` способна полностью остановить очистку БД, что приведет к катастрофическому раздуванию файлов базы на диске.

### В. Исчерпание пула соединений

Каждое «зависшее» соединение удерживает слот памяти. При достижении лимита `max_connections` новые пользователи просто не смогут зайти на сайт, получая ошибку `Too many connections`.

---

## 3. Варианты решения и защиты

Существует три профессиональных способа нивелировать эти риски во Flask-приложениях.

---

### Подход А. Автоматический откат через контекст Flask (`@app.teardown_appcontext`)

Этот метод используется, когда необходимо сохранить стандартное транзакционное поведение `psycopg2`, но гарантировать безопасность веб-запросов. Мы используем хук жизненного цикла Flask, который принудительно «зачищает» транзакции после того, как ответ улетел пользователю.

#### декоратор @app.teardown_appcontext

Это встроенный хук жизненного цикла Flask. Он работает по следующей схеме:

```text
[Браузер] -> Запрос страницы /catalog
     │
     ▼
Flask вызывает роут catalog()
     │
     ▼
psycopg2 делает SELECT (Postgres открывает транзакцию и вешает LOCK)
     │
     ▼
Flask рендерит HTML и ОТПРАВЛЯЕТ ответ пользователю
     │
     ▼
[В самом конце, когда ответ уже улетел] 
Flask автоматически вызывает функцию shutdown_session() 
     │
     ▼
Выполняется connection.rollback() -> Транзакция в Postgres закрывается!
```

Если в будущем появится роут, который что-то записывает в базу (например, оформление заказа или добавление машины через админку), вы сделаете connection.commit() прямо внутри этого роута после выполнения INSERT/UPDATE. А этот хук в конце запроса просто подстрахует: убедится, что транзакция в любом случае закрыта, и снимет локи.

#### Пример реализации

```python
from flask import Flask
from connect_db import connection

app = Flask(__name__)

@app.teardown_appcontext
def shutdown_session(exception=None):
    """Вызывается автоматически в самом конце жизненного цикла запроса"""
    if connection:
        try:
            # Закрывает неявную транзакцию, снимая все блокировки (LOCK)
            connection.rollback() 
        except Exception:
            pass

```

#### Сфера применения

* Проекты средней сложности на чистом SQL, где периодически требуются ручные транзакции (`BEGIN/COMMIT`) внутри бизнес-логики.
* Приложения, где критически важно контролировать момент закрытия ресурсов.

#### Ограничения

* **Избыточность на чтение:** База данных все равно тратит ресурсы на создание `BEGIN` и `ROLLBACK` для каждого посещения страницы, даже если там не было изменений данных.
* **Человеческий фактор:** Если в коде одного из роутов соединение упало или зависло до фазы `teardown`, транзакция все равно заблокирует таблицу на некоторое время.

### Подход Б. Режим Autocommit (`connection.autocommit = True`)

Перевод соединения в режим `autocommit` полностью отключает неявный вызов `BEGIN` со стороны `psycopg2`.

Каждый одиночный запрос (будь то `SELECT`, `INSERT` или `DELETE`) отправляется в базу независимо, изолированно и сразу же автоматически фиксируется СУБД.

#### Пример реализации (`connect_db.py`)

```python
import psycopg2

connection = psycopg2.connect(...)
# Полное отключение скрытых BEGIN:
connection.autocommit = True 

```

#### Сфера применения

* Идеально для каталогов, витрин, контентных сайтов и CRUD-систем, где 95% операций — это одиночные `SELECT`-ы или простые атомарные записи.
* Рекомендуется на этапе активной разработки UI, чтобы накатывание `.sql` файлов структуры не конфликтовало с запущенным локально Flask.

#### Ограничения

* **Потеря атомарности для цепочек запросов:** Если вам нужно провести финансовую транзакцию (списать баланс `AND` создать заказ), режим `autocommit` выполнит их как два независимых шага. Если второй шаг упадет, первый не откатится сам.
* **Решение:** Для многошаговых операций внутри Python-кода придется открывать транзакцию **явно вручную**, используя контекстные менеджеры:

```python
# Явное создание безопасной транзакции в режиме autocommit:
with connection:
    with connection.cursor() as cursor:
        cursor.execute("UPDATE accounts SET balance = balance - 100 WHERE id = 1;")
        cursor.execute("INSERT INTO orders ...")
# При выходе из блока with произойдет автоматический COMMIT, при ошибке — ROLLBACK.

```

---

### Подход В. Переход на ORM (Flask-SQLAlchemy / SQLAlchemy)

По мере роста проекта ручное управление `psycopg2` становится слишком трудоемким. Промышленным стандартом является использование SQLAlchemy, которая берет на себя всю рутину по управлению сессиями базы данных.

#### Пример концепции

```python
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy(app)

# SQLAlchemy сама управляет транзакциями. 
# По окончании запроса она автоматически делает rollback/commit в зависимости от статуса.

```

#### Сфера применения

* Крупные enterprise-проекты с комплексной бизнес-логикой.
* Команды разработки, где важно стандартизировать работу с данными без написания «сырого» SQL.

#### Ограничения

* Высокий порог входа для гибкой оптимизации сложных `JOIN`-запросов.
* Дополнительное архитектурное наслоение (оверхед по памяти на маппинг объектов).

---

## Резюме: Что выбрать?

1. Для **витрин данных и каталогов** (как проект *DriveX*) оптимальным, самым чистым и безопасным выбором является **Подход Б (`autocommit = True`)**. Он полностью устраняет проблему блокировки таблиц при редактировании SQL-структуры в VS Code и не нагружает код лишними хуками.
2. Если в будущем проект начнет обрастать сложными транзакционными цепочками (например, полноценная платежная система), имеет смысл либо точечно использовать блоки `with connection:` поверх `autocommit`, либо внедрять **Подход А** вместе с пулом соединений (`Psychopg2 Connection Pool`).
