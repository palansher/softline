## Справка по SQL: Конструкция RETURNING в PostgreSQL## 1. Зачем нужна эта функция?

Обычно при изменении данных (INSERT, UPDATE, DELETE) база данных возвращает только технический ответ: «ОК, затронуто N строк».
Если приложению нужно узнать, какие именно данные записались (например, сгенерированный базой id или дефолтная дата), разработчикам приходилось делать два запроса: сначала изменять данные, а потом делать SELECT.
Конструкция RETURNING превращает команды изменения данных в подобие SELECT. Она позволяет одним запросом изменить данные и сразу вернуть их результат в приложение.

## Главная польза

- Производительность: Минус один лишний запрос к базе данных (экономия сетевого времени).
- Атомарность: Вы гарантированно получаете данные той строки, которую только что изменили.
- Удобство: Не нужно запрашивать SELECT MAX(id) или currval(), чтобы узнать последний созданный ID.

------------------------------

## 2. Ключевые примеры использования## А. Получение ID при вставке (INSERT)

Самый частый сценарий. Когда id генерируется автоматически (через SERIAL или IDENTITY), нам нужно узнать его сразу после вставки.

```
INSERT INTO users (name, email) VALUES ('Алексей', '<alex@example.com>')
RETURNING id, created_at;
```

- Что делает: Создает пользователя и сразу возвращает его новый id и системное время создания.

## Б. Возврат всей строки целиком (RETURNING *)

Если после обновления или вставки вам нужны вообще все поля строки, используйте звездочку.

```
UPDATE products SET price = price *1.05 WHERE category = 'Электроника'
RETURNING*;
```

- Что делает: Поднимает цены на 5% и выводит список всех измененных товаров со всеми их свойствами.

## В. Получение удаленных данных (DELETE)

Иногда перед удалением строки нужно сохранить её данные (например, для архива или вывода пользователю сообщения «Товар [Имя] успешно удален»).

```
DELETE FROM tasks WHERE status = 'completed'
RETURNING id, title;
```

- Что делает: Очищает таблицу от выполненных задач, но возвращает список их названий и ID.

------------------------------

## 3. Коротко о главном (Шпаргалка)

- RETURNING пишется в самом конце SQL-запроса.
- Работает с INSERT, UPDATE, DELETE и MERGE.
- Можно возвращать как отдельные столбцы (RETURNING id, name), так и всю строку (RETURNING *).
- С результатами RETURNING в коде приложения (Python, Node.js, Go) работают точно так же, как с результатами обычного SELECT.

💡 Важное примечание для будущего: Начиная с PostgreSQL 18, эту функцию улучшили. Теперь при обновлении данных (UPDATE) внутри RETURNING можно использовать ключевые слова OLD.имя_поля и NEW.имя_поля, чтобы увидеть состояние строки «до» и «после» одним запросом.

## Использование PostgreSQL 18+ RETURNING (OLD/NEW) в Python

подробно расписана работа с новыми псевдонимами OLD и NEW (из PostgreSQL 18+), а также показано, как правильно принимать эти данные в Python через библиотеку psycopg (современная третья версия psycopg3)

Начиная с PostgreSQL 18, при обновлении данных (UPDATE) мы можем использовать префиксы OLD. и NEW. внутри блока RETURNING. Это позволяет в рамках одного запроса забрать состояние строки до изменения и после него.

### Подготовка таблицы для примеров

```sql
CREATE TABLE inventory (
    id SERIAL PRIMARY KEY,
    item_name VARCHAR(100),
    quantity INT,
    last_updated TIMESTAMP DEFAULT NOW()
);
INSERT INTO inventory (item_name, quantity) VALUES ('Смартфоны', 50);
```

### Пример 1: Фиксация разницы (изменения) в Python

Сценарий: Пользователь покупает товар. Нам нужно уменьшить остаток на складе, вычислить дельту и сразу вернуть её в Python для логирования или вывода пользователю.

```Python
import psycopgfrom psycopg.rows import dict_row

# Подключение к базе данных (используем dict_row для удобного доступа по ключам)conn_info = "dbname=mydb user=postgres password=secret host=localhost"

with psycopg.connect(conn_info, row_factory=dict_row) as conn:
    with conn.cursor() as cur:

        # SQL-запрос с использованием новинок PostgreSQL 18 (OLD и NEW)
        query = """
            UPDATE inventory 
            SET quantity = quantity - %s 
            WHERE item_name = %s
            RETURNING 
                item_name,
                OLD.quantity AS stock_before,
                NEW.quantity AS stock_after,
                (OLD.quantity - NEW.quantity) AS quantity_sold;
        """
        
        # Выполняем запрос (продаем 5 смартфонов)
        cur.execute(query, (5, 'Смартфоны'))
        
        # Так как есть RETURNING, мы забираем результат через fetchone() как в SELECT
        result = cur.fetchone()
        
        if result:
            print(f"Товар: {result['item_name']}")
            print(f"Было на складе: {result['stock_before']}")
            print(f"Стало на складе: {result['stock_after']}")
            print(f"Успешно продано: {result['quantity_sold']} шт.")

```

Что выдаст Python в консоль:

```
Товар: Смартфоны
Было на складе: 50
Стало на складе: 45
Успешно продано: 5 шт.
```

### Пример 2: Умный аудит (Было / Стало) всей строки

Если вам нужно логировать в таблицу аудита (history log) полные изменения объекта, можно вернуть старую и новую строки целиком, используя синтаксис OLD.*и NEW.*.

```Python
import psycopgimport json
with psycopg.connect("dbname=mydb user=postgres host=localhost") as conn:
    with conn.cursor() as cur:

        # Превращаем всю старую и всю новую строку в JSON прямо на стороне PostgreSQL
        query = """
            UPDATE inventory 
            SET quantity = 100, item_name = 'Флагманские Смартфоны'
            WHERE id = 1
            RETURNING row_to_json(OLD.*) AS before_changes, row_to_json(NEW.*) AS after_changes;
        """
        
        cur.execute(query)
        row = cur.fetchone()
        
        if row:
            # В старых версиях PG пришлось бы делать предварительный SELECT.
            # Теперь мы получили оба состояния за 1 атомарный проход базы.
            old_state = row[0]
            new_state = row[1]
            
            print("Исторический лог успешно сформирован:")
            print(f"СТАРОЕ СОСТОЯНИЕ: {old_state}")
            print(f"НОВОЕ СОСТОЯНИЕ: {new_state}")
            
            # Здесь можно в один клик записать эти JSON в таблицу логов audit_logs
```

### Главные правила для разработчика на Python

   1. Забываем про conn.commit() перед чтением: Раньше, если вы делали UPDATE, нужно было закоммитить транзакцию, а потом делать SELECT. С RETURNING вы сначала забираете данные через cur.fetchone() или cur.fetchall(), и только потом вызываете conn.commit().
   2. Обработка пустых результатов: Если условие WHERE не выполнилось (строка не найдена), UPDATE ... RETURNING не вызовет ошибку. Он просто вернет 0 строк. Поэтому в Python всегда проверяйте if result is not None:.
   3. Экономия на блокировках строк: Использование OLD/NEW в RETURNING происходит внутри одной транзакции и одной операции. Это полностью исключает ситуацию (Race Condition), когда между вашим SELECT и UPDATE другой параллельный Python-процесс успел изменить эту же строку.
