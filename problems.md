## .venv в проекте

Чтобы терминал в VS Code автоматически активировал виртуальное окружение (.venv) при каждом открытии, вам нужно настроить сам редактор. VS Code умеет автоматически запускать скрипт активации для новых терминалов.

### Шаг 1. Создайте .venv внутри проекта (если еще не создано)

Убедитесь, что вы находитесь в папке проекта, и выполните команду в терминале:

`python3 -m venv .venv`

### Шаг 2. Настройте VS Code для автоактивации

Вам нужно указать VS Code использовать именно этот интерпретатор Python.

   1. Нажмите комбинацию клавиш Ctrl + Shift + P (или F1), чтобы открыть палитру команд.
   2. Введите и выберите: Python: Select Interpreter (Python: Выбрать интерпретатор).
   3. В списке выберите интерпретатор с пометкой (.venv) или пути вида ./.venv/bin/python. (не Global)

### Шаг 3. Включите автоактивацию в настройках

Обычно VS Code включает это сам, но стоит проверить:

   1. Откройте настройки (Ctrl + ,).
   2. В строке поиска введите: python.terminal.activateEnvironment.
   3. Убедитесь, что стоит галочка (значение True).

### Шаг 4. Проверьте результат

Закройте текущий терминал (нажмите на значок мусорной корзины в углу панели терминала) и откройте новый через меню Terminal -> New Terminal (`Ctrl + Shift + ``).

Приглашение командной строки изменится, и в самом начале появится префикс окружения:

```bash
(.venv) ✔ ~/code/learn-python [main|✚ 1…2] 03:46 $ 
```

## С терминалом PowerShell Windows

Вы столкнулись с очень известной и коварной особенностью консоли Windows (компонента conhost или встроенного терминала VSCode), которая называется «Режим быстрого редактирования» (QuickEdit Mode).

Когда вы кликаете мышкой внутрь терминала, Windows полностью приостанавливает (замораживает) выполнение скрипта, чтобы вы могли выделить текст. Для сетевого сокета это выглядит так, будто сервер мгновенно «умер»: тайм-ауты истекают, и операционная система разрывает соединение, выдавая ошибку клиенту. А когда вы нажимаете клавиши (например, Enter или Ctrl+C, чтобы отменить выделение), скрипт «просыпается», перехватывает это нажатие как сигнал прерывания и выводит сообщение [Сервер] Остановлен пользователем.
Вот несколько способов, как сделать работу в терминале VSCode на Windows 11 удобной и избавиться от этого эффекта.

### Способ 1. Использовать современный Windows Terminal (Рекомендуется)

В Windows 11 встроен новый, современный терминал, где клик мышкой не блокирует выполнение программ. Чтобы VSCode использовал именно его:

   1. Откройте настройки VSCode (Ctrl + ,).
   2. В строке поиска введите: Terminal › Integrated: Default Profile: Windows.
   3. Убедитесь, что там выбран PowerShell (а не классический Command Prompt).
   4. Запускайте код кнопкой Run Python File.

### Способ 2. Как правильно снимать «заморозку» мышкой

Если вы случайно кликнули в терминал и увидели, что на панели терминала сверху появилось слово [Выделение] (или Select), либо код просто замер:

- Нажмите клавишу Esc или Enter.
- Это снимет выделение текста, и Windows мгновенно возобновит работу вашего сервера без падения.

### Способ 3. Отключить QuickEdit в системе (Для старых консолей)

Если терминал открывается в отдельном системном окне cmd/powershell:

   1. Кликните правой кнопкой мыши по верхней панели (заголовку) окна терминала.
   2. Выберите Свойства (Properties).
   3. На вкладке «Общие» (Options) снимите галочку с пункта «Выделение мышью» (QuickEdit Mode).
   4. Нажмите ОК.

------------------------------

### Почему при первом запуске всё работало?

При первом старте терминал был «чистым». При повторных запусках (особенно через команду Run in dedicated terminal) VSCode повторно использует то же окно. Если в процессе клика курсор попал на старый текст логов, Windows автоматически инициировала режим выделения, заблокировав сетевой поток Python.
Чтобы мы могли проверить, нет ли в самом коде сервера скрытой уязвимости к таким прерываниям, пришлите:

- Актуальный текст вашего файла server.py из папки hw (особенно ту часть, где перехватывается исключение для вывода строки Остановлен пользователем)

## С терминалом PowerShell Linux

### Появление символа ^C (KeyboardInterrupt) при активации venv в терминале

Появление символа ^C (KeyboardInterrupt) без вашего участия — это известная проблема интеграции VS Code и расширения Python при работе по Remote-SSH. [1, 2]

Когда создается новый терминал, расширение Python пытается отправить строку активации source .../activate.

Однако встроенная интеграция оболочки VS Code (Shell Integration) в этот же момент инициализирует саму себя.

Происходит конфликт таймингов: расширение Python пугается, что терминал "занят" или не успел ответить, и автоматически посылает в скрытый буфер Ctrl+C, чтобы сбросить строку, а уже затем выполняет команду заново. Из-за этого первая автоматическая попытка всегда прерывается. [1, 3, 4]

Чтобы исправить это раз и навсегда для вашей конфигурации Ubuntu 24.04, выполните следующие шаги.

#### Вариант 1. Отключить конфликтную настройку (Рекомендуется)

Часто проблему решает отключение функции Shell Integration для терминала, которая конфликтует с расширением Python. [3]

   1. Откройте настройки VS Code (Ctrl + ,).
   2. В строке поиска введите: terminal.integrated.shellIntegration.enabled.
   3. Переключите вкладку на Remote [SSH: ...] (как на вашем втором скриншоте), чтобы изменить настройку только для сервера.
   4. Снимите галочку (выключите её).
   5. Убейте терминал через корзину и откройте новый. [3, 5]

------------------------------

#### Вариант 2. Железная активация через .bashrc на сервере

Если вы не хотите отключать Shell Integration, можно полностью запретить VS Code самому слать команды активации, и переложить эту задачу на саму операционную систему Ubuntu.

Шаг A. Выключите автоактивацию в VS Code

   1. Откройте настройки (Ctrl + ,), перейдите на вкладку Remote [SSH: ...].
   2. Найдите python.terminal.activateEnvironment.
   3. Снимите галочку (отключите её). Теперь VS Code вообще не будет пытаться слать команды в терминал при старте. [1]

Шаг B. Пропишите активацию в профиль пользователя Linux

- 1 В терминале VS Code введите команду для редактирования файла конфигурации вашей оболочки:

   `nano ~/.bash_profile`

   (Если файла не существует, nano откроет пустой экран — это нормально).

- 2 Опуститесь в самый низ (или вставьте в пустой файл) следующий скрипт:

```bash
   # Автоматическая активация .venv при входе в папку проекта
   if [ -d "$HOME/code/learn-python/.venv" ]; then
       source "$HOME/code/learn-python/.venv/bin/activate"
   fi
```

- 3 Сохраните изменения: нажмите Ctrl + O, затем Enter. Выйдите из редактора: Ctrl + X.

Удалите терминал через корзину и создайте заново (Ctrl + \``).

Поскольку за запуск скрипта теперь отвечает сам Linux в момент инициализации сессии Bash, никаких фантомных ^Cбольше не возникнет, и в строке сразу появится нужный вам префикс(.venv)`.

Если вы хотите проверить, применились ли изменения, или у вас возникли сложности с сохранением файла в nano, укажите нужный вариант:

## GIT CRLF/LF

В Git есть встроенная функция автоматической конвертации строк. Когда вы делаете git clone на Linux-сервере, Git проверяет настройки и, если нужно, сам превращает Windows-формат (CRLF) в Linux-формат (LF) прямо во время скачивания файлов [1].

```bash
git config --global core.autocrlf input
Используйте код с осторожностью.Что это делает: Эта опция гарантирует, что при скачивании репозитория на Linux файлы будут автоматически приводиться к родному стилю LF [1].
```

## pip install psycopg2 - Ошибка Error: pg_config executable not found

Ошибка Error: pg_config executable not found возникает потому, что пакет psycopg2 пытается скомпилироваться из исходного кода, но в вашей Ubuntu 24.04 не установлены системные библиотеки PostgreSQL (в частности, утилита pg_config).

У вас есть два пути решения этой проблемы.

### Вариант 1. Установить бинарную версию (Самый быстрый и простой)

Разработчики psycopg2 выпускают готовую сборку, которой не нужны системные библиотеки для компиляции. Для домашней разработки и обучения это идеальный вариант.
Просто выполните в терминале:

```bash
pip install psycopg2-binary
```

Вам больше не нужно ставить чистый psycopg2.

Пакет psycopg2-binary предоставляет точно такой же синтаксис, те же функции и модули.
В коде вы будете писать привычное: import psycopg2.

### Вариант 2. Установить системные зависимости в Ubuntu (Если нужен именно psycopg2)

Если для проекта критически важно использовать именно чистый psycopg2 (например, для продакшена), вам необходимо установить пакет разработки PostgreSQL в саму операционную систему через apt.

Выполните по очереди следующие команды:

```bash
   # 1. Обновите списки пакетов:
   
   sudo apt update
   
   # 2. Установите компилятор и библиотеки PostgreSQL:
   
   sudo apt install build-essential libpq-dev
   
   # 3. Снова запустите установку в вашем виртуальном окружении:
   
   pip install psycopg2
```
