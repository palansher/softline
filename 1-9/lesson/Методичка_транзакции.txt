
Транзакция - это набор запросов, который требует либо подтверждения, либо отмены. Транзакция либо выполняется 
полностью либо не выполняется совсем. Транзакция после выполнения должна оставаться в консистентном состоянии (непротиворечивом состоянии).
То есть выполняется группа запросов и если хотя бы один запрос выполняется некорректно мы останавливаем выполнение всех запросов и отменяем
их.

Уровни изоляции транзакции решают проблему обращения к общим данным.

Есть риск, что в один и тот же момент времени к данным обращаются разные сервисы. Существует 4 уровня изоляции транзакции.

1) READ UNCOMMITED(грязное чтение) - это чтение одной транзакции данных, которые были изменены другой транзакцией, но не были зафиксированы
другой транзакцией.

2) READ COMMITED(установлен по умолчанию) - транзакции работают только с теми данными, которые зафиксированы.\
Все транзакции работают с последними изменениями

BEGIN;
SHOW TRANSACTION_ISOLATION;


Любая транзакция начинается с оператор BEGIN;
Можно явно закрыть транзакцию с помощью оператора END;

3) REPEATABLE READ - этот уровень изоляции транзакции видит только те данные,
которые зафиксированы до начала транзакции.
Данный уровень видит только первый коммит, последующие игнорируются,
 поэтому выигрываем в скорости обращения к данным.
Пример налоговая и интернет-магазин.

4) SERIALIZABLE - транзакции выполняются последовательно независимо друг от друга. То есть если одна транзакция к данным
обратилась, то другие транзакции уже не имеют доступ к этим данным, пока первая транзакция не завершит работу. Цель -
это максимальная безопасность. Скорость самая медленная.

ACID - требования к транзакции
A (атомарность транзакции) - транзакция выполняется либо полностью либо не выполняется совсем
C (консистентность) - данные после выполнения транзакции должны оставаться в непротиворечивом состоянии.
I (изолированность) - транзакции работают на основе уровня изоляции транзакции
D (устойчивость к изменениям) - за счет журнала транзакции после любого сбоя можно продолжить выполнение транзакции

Для работы с транзакциями мы используем 4 основных оператора: BEGIN, END, COMMIT, ROLLBACK

1) CREATE TABLE clients(client_id serial PRIMARY KEY,fio varchar(20),balance int);
   INSERT INTO clients(fio,balance) VALUES('Иванов',1000),('Петров',1000),('Сидоров',1000)

2) Пример с транзакцией
BEGIN;
	INSERT INTO clients(fio,balance) VALUES('Романов',2000);
COMMIT;
select * from clients;

Задание
1) Внутри транзакции удалить запись с клиентом №1 и вывести оставшиеся записи
2) Отменяем вышесозданные действия, чтобы клиент не был удален

Пример:

BEGIN;
	UPDATE clients SET balance = balance + 500 WHERE client_id = 1;
	UPDATE clients SET balance = balance - 500 WHERE client_id = 2;
	DELETE FROM clients WHERE client_id=1;
    COMMIT;
END;
select * from clients;

Взаимодействие с пользователем

1) Создание юзера
CREATE USER petr  WITH PASSWORD '123456'

2) Пример как дать все привилегии:
GRANT ALL privileges ON DATABSE 'test_db' TO petr
Права только на чтение данных:
\c test_db
GRANT pg_read_all_data TO petr
GRANT pg_write_all_data TO petr

3) Редактирование пользователя

   3.1 Смена пароль: ALTER USER postgres PASSWORD 'новый пароль'
   3.2 Забираем права у юзера: REVOKE ALL privileges ON ALL TABLES IN SCHEMA public FROM "petr"
   3.3 Помимо ALL privileges можно выдать права только на некоторые операции:
      GRANT SELECT,UPDATE,INSERT ON ALL TABLES IN public TO "petr"
